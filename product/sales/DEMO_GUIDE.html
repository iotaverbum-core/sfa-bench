GroundLedger Demo Guide

A plain-language, step-by-step manual for the person selling it. If you can paste a line into a terminal, you can run this demo.

Commands are written for Windows (they also work on Mac/Linux). If python says "not found", try python3. There is nothing to install - no accounts, no keys, no internet.

1. The 30-second idea

Companies put AI assistants in front of customers to answer questions from their documents. Sometimes the assistant makes things up - it cites a section that doesn't exist, or states a number the source contradicts. That's a hallucination, and it's a real risk in regulated industries.

GroundLedger checks every answer against the documents it should have used, and keeps a permanent, tamper-evident record you can prove to an auditor. It runs on the customer's own computers, so their documents never leave.

2. The story your demo tells (three beats)

1. We catch the bad answers. Four sample answers go in; one is fine, three are wrong - and it says exactly why.

2. We turn that into evidence. Each verdict is sealed into a report the customer's auditor can re-check themselves.

3. Nobody can quietly fudge it. We edit a failed answer to look like it passed - the system catches it: TAMPER DETECTED.

3. Words you'll hear

  • Grounded - the answer is backed by the source documents.
  • Citation - the document/section the answer points to (e.g. clause_3a).
  • Fabricated citation - it points to a section that doesn't exist.
  • Contradiction - it states a value that disagrees with the source.
  • Receipt - a sealed record of one answer's verdict (like a store receipt you can't secretly change).
  • Ledger - the chained list of all receipts; nothing can be deleted or reordered without it showing.
  • Groundedness rate - the percentage of answers that were grounded.
  • Tamper-evident - edits are detected (not the same as "impossible to edit" - be honest).
  • Replay / attest - re-checking the whole record to confirm the verdicts are real ("prove it yourself").

4. One-time setup

Check you have Python 3.11+:

python --version

You should see Python 3.11.x or higher (else install from python.org and tick "Add to PATH").

Open a terminal inside the sfa-bench folder (Windows: type cmd in the folder's address bar and press Enter).

5. Run the demo (the main event)

python -m product.demo

You'll see: (A) four answers checked - one PASS, three FAIL with the reason; (B) a report summary ("1 of 4 answers (25%) grounded") with findings ranked Critical / High / Medium; (C) two saved files:

report : product/data/demo/report.html   (open in a browser)
bundle : product/data/demo/bundle.json    (verifies offline)

If you got here, you've run the demo - that's 90% of it.

6. The moment that sells it

Now an insider quietly edits a sealed failure to look like a pass...
  re-attestation: TAMPER DETECTED

Someone tried to cheat - to make a bad answer look good. The system re-checked and caught it.

"Your evals live in a dashboard you could edit. This is a record your auditor re-runs themselves - and if anyone changes it, it shows. That's the difference between a metric and evidence."

7. Show the pretty report in a browser (optional wow)

python -m product.groundledger.ingest product/examples/batch.jsonl --tenant demo-tenant --data-root product/data
python -m product.groundledger.api

Open http://127.0.0.1:8000/, type the key demo-key, click "View audit report". Press Ctrl+C in the terminal to stop. Everything runs on your computer - nothing goes to the internet.

8. Prove it's trustworthy

python -m product.groundledger.verification

You'll see reproducibility: PASS, tamper detection: PASS, final status: VERIFIED.

"The same inputs always produce the exact same result - I can prove it, and so can you. That's why an auditor will accept it."

9. Run it on real answers

Put the answers in a JSONL or CSV file (see product/examples/batch.jsonl for the shape), then:

python -m product.groundledger.ingest their-answers.jsonl --tenant acme --data-root product/data

It prints how many were grounded and the findings. This is exactly what a paid pilot does - on ~150-300 of their answers.

10. Running a live demo call (~15 min)

Full script: product/sales/calls/demo-script-15min.md. Short version: frame the problem → run python -m product.demo → the tamper moment → show the browser report → close ("let's run this on ~200 of your real answers"). Practice the run 3-4 times until it feels automatic.

11. If something goes wrong

  • "python not recognized": try python3; else reinstall Python with "Add to PATH".
  • "No module named product": you're in the wrong folder - be inside sfa-bench (you should see product and sfa).
  • Browser says "API key required": add the key - go to http://127.0.0.1:8000/ and type demo-key.
  • Unexpected "TAMPER DETECTED": the demo leaves one answer tampered on purpose. Delete the product/data folder to start clean.
  • Reset everything: delete the product/data folder (only demo/run data lives there).

13. What NOT to say (stay honest)

  • Not "tamper-proof" → say "tamper-evident" (edits are detected).
  • Not "certified/compliant" → say "audit-ready evidence for your own compliance process."
  • Not "catches every hallucination" → say "reliably catches fabricated citations and contradictions, and records everything."
  • Strongest on answers that cite their sources; free-text is supported but more conservative.

Being straight about the limits is what makes a skeptical buyer trust the rest. Full boundary: product/TRUST_MODEL.md.
